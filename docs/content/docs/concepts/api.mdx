When you create a Better Auth Ruby instance, it exposes two entry points:

- `auth.handler` / `auth.call(env)` for Rack requests.
- `auth.api` for server-side calls to the same endpoint implementations.

Every core endpoint and plugin endpoint is exposed through `auth.api` using a
Ruby `snake_case` method name.

```ruby title="config/auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  base_url: "http://localhost:3000",
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  email_and_password: {
    enabled: true
  }
)
```

## Server-Side Calls

Call endpoints with `body`, `headers`, `query`, and `params` keys.

```ruby title="app/services/auth_examples.rb"
session = auth.api.get_session(
  headers: {
    "cookie" => request.get_header("HTTP_COOKIE").to_s
  }
)

sign_in = auth.api.sign_in_email(
  body: {
    email: "john@example.com",
    password: "password123"
  },
  headers: {
    "user-agent" => request.user_agent.to_s,
    "x-forwarded-for" => request.ip.to_s
  }
)

verified = auth.api.verify_email(
  query: {
    token: params.fetch(:token)
  }
)
```

| Endpoint key | Ruby API method | HTTP path |
| --- | --- | --- |
| `sign_up_email` | `auth.api.sign_up_email` | `/api/auth/sign-up/email` |
| `sign_in_email` | `auth.api.sign_in_email` | `/api/auth/sign-in/email` |
| `get_session` | `auth.api.get_session` | `/api/auth/get-session` |
| `link_social` | `auth.api.link_social` | `/api/auth/link-social` |
| `get_access_token` | `auth.api.get_access_token` | `/api/auth/get-access-token` |

## Headers And Rack Responses

By default, `auth.api` returns the endpoint response object directly.

Use `return_headers: true` when you need response headers such as
`set-cookie`.

```ruby title="app/services/create_user.rb"
result = auth.api.sign_up_email(
  return_headers: true,
  return_status: true,
  body: {
    email: "john@example.com",
    password: "password123",
    name: "John Doe"
  }
)

status = result[:status]
headers = result[:headers]
body = result[:response]
session_cookie = headers["set-cookie"]
```

Use `as_response: true` when you need the Rack response tuple.

```ruby title="app/controllers/sessions_controller.rb"
status, headers, body = auth.api.sign_in_email(
  as_response: true,
  body: {
    email: "john@example.com",
    password: "password123"
  }
)

response.status = status
headers.each { |name, value| response.set_header(name, value) }
response.write(body.join)
```

## Error Handling

Server-side API calls raise `BetterAuth::APIError` unless `as_response: true` is
used.

```ruby title="app/services/sign_in.rb"
begin
  auth.api.sign_in_email(
    body: {
      email: "john@example.com",
      password: "wrong-password"
    }
  )
rescue BetterAuth::APIError => error
  Rails.logger.info(
    "Better Auth failed: #{error.status_code} #{error.message}"
  )
end
```

## Rack Requests

The same auth object is a Rack app.

```ruby title="config.ru"
require "better_auth"

auth = BetterAuth.auth(
  base_url: "http://localhost:9292",
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  email_and_password: {
    enabled: true
  }
)

map "/api/auth" do
  run auth
end
```

Rails apps normally mount the adapter instead:

```ruby title="config/routes.rb"
Rails.application.routes.draw do
  better_auth
end
```

## Rate Limit Boundary

Rate limiting runs in the Rack router. Direct `auth.api` calls are server-side
function calls and are not rate limited.

If an external user can trigger a server action, controller action, job, or RPC
method that calls `auth.api`, rate limit that public entry point separately.

## Not Supported

- JavaScript `Headers` or `Response` objects. Ruby uses hashes and Rack response
  tuples.
- TypeScript endpoint inference. Ruby endpoint methods are generated at runtime.
- The upstream `better-call` API surface as a public Ruby dependency.
