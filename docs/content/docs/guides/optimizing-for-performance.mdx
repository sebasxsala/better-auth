In this guide, we'll go over some of the ways you can optimize your Ruby application for a more performant Better Auth setup.

## Cookie Cache

Calling your database every time a session is looked up isn't ideal, especially if sessions don't change frequently. Cookie caching handles this by storing session data in a short-lived, signed cookie similar to how JWT access tokens are used with refresh tokens.

To turn on cookie caching, set `session.cookie_cache` in your auth config:

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  session: {
    cookie_cache: {
      enabled: true,
      max_age: 5 * 60, # Cache duration in seconds
      strategy: "jwe"  # "jwe", "jwt", or "compact"
    }
  }
)
```

Read more about [cookie caching](/docs/concepts/session-management#cookie-cache).

## Framework Caching

Because Better Auth Ruby is Rack-based, you can wrap auth API calls with any caching layer your framework provides or with Rack middleware.

**Rails example using fragment caching:**

```ruby title="app/controllers/users_controller.rb"
class UsersController < ApplicationController
  def index
    users = Rails.cache.fetch("auth_users", expires_in: 1.hour) do
      auth.api.list_users[:users]
    end
    render json: users
  end
end
```

**Rack::Cache example:**

```ruby title="config.ru"
use Rack::Cache,
    metastore: "file:/var/cache/rack/meta",
    entitystore: "file:/var/cache/rack/body"
run app
```

Cache responses that do not change often (such as public provider metadata or user lists) to reduce repeated auth API calls.

## Background Tasks

On serverless platforms or multi-process servers, you can improve response times by deferring non-critical work (cleanup, analytics, rate limit updates, email sending) to run after the response is sent. Configure `advanced.background_tasks` and use `ctx.context.run_in_background` in hooks.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    background_tasks: {
      handler: ->(task) { Thread.new { task.call } }
    }
  },
  hooks: {
    after: ->(ctx) {
      if ctx.path == "/sign-up/email"
        ctx.context.run_in_background(-> { Analytics.log_sign_up(ctx.context.new_session&.dig("user", "id")) })
      end
    }
  }
)
```

See the [Hooks documentation](/docs/concepts/hooks#runinbackground) for more examples.

## Database Optimizations

Optimizing database performance is essential to get the best out of Better Auth.

### Recommended Fields to Index

| Table         | Fields                     | Plugin       |
| ------------- | -------------------------- | ------------ |
| users         | `email`                    |              |
| accounts      | `user_id`                  |              |
| sessions      | `user_id`, `token`         |              |
| verifications | `identifier`               |              |
| invitations   | `email`, `organization_id` | organization |
| members       | `user_id`, `organization_id` | organization |
| organizations | `slug`                     | organization |
| passkey       | `user_id`                  | passkey      |
| two_factor    | `secret`                   | two_factor   |

<Callout>
  In Better Auth Ruby all database fields use `snake_case` by convention. If you are writing raw SQL or migrations, use `user_id`, `organization_id`, etc.
</Callout>
