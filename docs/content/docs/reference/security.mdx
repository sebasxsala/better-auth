title: Security

Better Auth Ruby keeps the upstream security model where it applies to Rack and Ruby runtimes: secure password hashing, signed cookies, OAuth state validation, origin checks, callback URL validation, session revocation, and rate limiting.

## Password Hashing

Passwords are hashed with `scrypt` by default.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  password_hasher: :scrypt
)
```

`bcrypt` is also supported.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  password_hasher: :bcrypt
)
```

## Sessions

Sessions are identified by a signed `session_token` cookie and resolved on the server. With a database adapter, session records are persisted in the database. Without a database, the Ruby port enables stateless defaults and stores session cache data in signed or encrypted cookies.

Sessions expire after seven days by default.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  session: {
    expires_in: 60 * 60 * 24 * 7,
    update_age: 60 * 60 * 24,
    fresh_age: 60 * 60 * 24
  }
)
```

`fresh_age` protects sensitive routes such as password changes, email changes, and account deletion. Set it to `0` only when you intentionally want to disable fresh-session checks.

See [Session Management](/docs/concepts/session-management).

## CSRF Protection

Better Auth Ruby rejects unsafe cross-origin requests in several layers.

1. Unsafe methods require a trusted `Origin` or `Referer` header when cookies are present.
2. `callbackURL`, `redirectTo`, `errorCallbackURL`, and `newUserCallbackURL` are validated against trusted origins.
3. Fetch Metadata headers block first-login CSRF attempts, such as cross-site navigations to sign-in routes before cookies exist.
4. `GET`, `HEAD`, and `OPTIONS` are treated as non-mutating by the origin-check middleware.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  base_url: "https://auth.example.com",
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  trusted_origins: [
    "https://app.example.com",
    "https://*.preview.example.com"
  ]
)
```

Relative redirect paths such as `/dashboard` are allowed where the route accepts relative callback URLs. Protocol-relative URLs such as `//evil.example` are not treated as safe relative paths.

## Disabling Security Checks

`advanced.disable_csrf_check` disables CSRF protection.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    disable_csrf_check: true
  }
)
```

`advanced.disable_origin_check` disables callback and redirect URL validation. For backward compatibility, setting it to `true` without also setting `disable_csrf_check` currently disables CSRF checks too and emits a deprecation warning.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    disable_origin_check: ["/custom-webhook"]
  }
)
```

<Callout type="warn">
These options remove protections against CSRF and open redirect attacks. Prefer adding the correct `trusted_origins` entry whenever possible.
</Callout>

## OAuth State

OAuth state protects sign-in and account-linking flows from CSRF and replay attacks. For stateless setups, the default account configuration stores state in cookies.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  database: nil,
  account: {
    store_state_strategy: "cookie",
    store_account_cookie: true
  }
)
```

If users repeatedly see state errors, check that the same `BETTER_AUTH_SECRET`, database, and secondary storage are shared across all deployed instances.

## Cookies

Better Auth Ruby cookies are `http_only`, `same_site: "lax"`, and scoped to `/` by default. Cookies are marked secure when `advanced.use_secure_cookies` is true, when `base_url` uses HTTPS, or when the app runs in production.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  base_url: "https://auth.example.com",
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    use_secure_cookies: true,
    cookie_prefix: "my-app"
  }
)
```

Cross-subdomain cookies are available when the auth server and application share a trusted parent domain.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  base_url: "https://auth.example.com",
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    cross_subdomain_cookies: {
      enabled: true,
      domain: "example.com"
    }
  },
  trusted_origins: ["https://app.example.com"]
)
```

See [Cookies](/docs/concepts/cookies).

## Rate Limiting

Rate limiting protects HTTP requests handled by the Rack router. It is enabled automatically in production and disabled by default in development and test.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  rate_limit: {
    enabled: true,
    window: 60,
    max: 100
  }
)
```

See [Rate Limit](/docs/concepts/rate-limit).

## IP Address Headers

Rate limiting and security checks use the request IP. Configure trusted headers only when your reverse proxy controls them and users cannot forge them.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    ip_address: {
      ip_address_headers: ["cf-connecting-ip", "x-forwarded-for"],
      disable_ip_tracking: false
    }
  }
)
```

## Trusted Proxy Headers

If the app runs behind a reverse proxy, `advanced.trusted_proxy_headers` allows Better Auth Ruby to infer request origins from `X-Forwarded-Host` and `X-Forwarded-Proto` when `base_url` is not configured.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    trusted_proxy_headers: true
  }
)
```

<Callout type="warn">
Only enable this when your proxy overwrites forwarded headers and strips any client-supplied values. Forged proxy headers can affect generated URLs, callbacks, and redirects.
</Callout>

## Reporting Vulnerabilities

Please do not report security issues in public issues. Follow the repository [Security Policy](https://github.com/sebasxsala/better-auth/blob/main/SECURITY.md) and include enough detail for maintainers to reproduce the issue.
